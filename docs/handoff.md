# 引き継ぎ（これまでの経緯と今後の展望）

## 1. 今の立ち位置

- 勉強会の **運営方針と全 5 回の構成はフィックス済み**（→ `docs/course_design.md`）。
- **各回で当日「具体的に何を話す・何を実装させるか」の詳細はこれから対話的に詰める段階**。
  第 1 回だけ骨子が固まっている（環境構築＝事前課題、data は当日 log-mel まで全部見せて宿題は `data.py` 整理）。
- 現行リポジトリ（`src/kws/`・`notebooks/`）は **旧方針のまま**（旧 6 回想定・解答公開前提・baseline が test acc≈0.90 と高い）。
  確定方針に沿って **刷新が必要**。

## 2. これまでの経緯（なぜこの大目標・構成・方針になったか）

- **当初案（旧版）**: torchaudio SPEECHCOMMANDS の穴埋め notebook ＋ 解答公開ブランチで「フレームワーク内部の理解」を
  狙う教材だった（→ `discontinued/plan.md`, `discontinued/curriculum.md`）。
- **大目標のピボット**: 大目標を **「PyTorch で動く学習・評価スクリプトを自分で書けるようになること」** に再定義。
  受講者は「ゼロから作る Deep Learning」で理論は履修済みだが実装経験が乏しい、という前提に合わせ、理論の再説明をやめ
  「自分で書く経験」に集中する方針にした。音声認識・torchaudio は題材＝手段と位置づけた。
- **丸写し対策**: 「解答公開 ＋ 穴埋め」だと丸写しで通過できる穴があったため、**解答非公開 ＋ 発表で動作確認例題を実演**に変更。
  詰まったらメンバー同士の相談＋主催者の時間外サポートで吸収する（部分解の配布などはしない）。
- **構成の確定**: 6 観点のサブエージェント（教育設計／運営／受講者／PyTorch 実務／コンペ／批判者）で 2 ラウンド多角評価した。
  最大の論点だった「学習ループ＋評価を 1 回に詰める第 4 回の過密」を、次の 2 点で解消した：
  1. 第 1 回も発表回にして、全 5 回を「前回モジュールの発表 → 次モジュールへ」の同型イテレーションにする
  2. 学習パートを **実装回（第 3 回）と Tips・深掘り回（第 4 回）に分離**する
- **baseline**: 既存 baseline は test acc≈0.90 と高く改善余地がないため、**弱 baseline（val 60〜70% / train 70〜80%）** に
  作り替える方針。test リークは教育目的なので許容する。
- **ドキュメント**: 読み手（リード）が異なるので分割管理する（下記）。

## 4. 何を読んで理解するか（この順で）

1. `docs/course_design.md` … 今回確定した目的・全 5 回構成・各回の概要・決定事項。**まずこれを読む**。
2. `docs/operation.md` … 運営の進め方（1 回のテンプレート・発表の運用・巡回・Tips の散らし方）。
3. `docs/sessions/index.md` … 各回の具体的内容のマネジメント（トップレベル箇条書き）。詰まり具合はここで把握。
4. プロジェクト memory `project_kws-tutorial.md` … 確定方針に至った経緯（解答非公開・大目標の再定義など）。
5. plan file（`~/.claude/plans/` 配下の本プロジェクトのもの）… 議論で確定した骨格の作業計画。
6. `discontinued/`（`plan.md` / `curriculum.md`）… 旧版。**現行方針ではない**ので参照のみ。

※ `docs/operation.md` は旧 6 回想定の記述が一部残っている可能性がある。構成・各回の確定情報は `docs/course_design.md` を正とする。

## 5. 今後の展望（次にやること）

### (a) 各回の具体的内容を tenk と対話しながら詰める

進め方は **トップレベル → 各回詳細** の 2 段階：

1. まず `docs/sessions/index.md`（トップレベル箇条書き）で各回に盛り込む内容をマネジメントし、**全体像を固める**。
2. 固まった回から `docs/sessions/sessionN_*.md` に **超詳細**を書く。
   ゴールは **「読めば誰でも再現性を持って資料（ノートブック）を作れる」レベル**（セル構成・説明の流れ・コードの骨子・
   動作確認例題と期待出力まで）。

詰めるべき論点：

- 事前学習で読ませる資料の指定（範囲・「これが言えれば OK」の最低ライン）
- 各回の当日の流れ（セル単位）／当日どこまで実装し、何を宿題に回すか／穴埋めの粒度
- 各モジュールの「これが動けば OK」動作確認例題の中身と期待出力
- 第 4 回の深掘り項目の見せ方（confusion matrix / per-class をここに置くか）
- 弱 baseline の具体 config（val 60〜70% / train 70〜80% になる設定。試走で確認する）
- 第 5 回 leaderboard の運用・評価指標

**第 1 回は決定済みの部分がある**（`docs/course_design.md` の「第 1 回」参照）。
残論点＝「読ませる資料の指定」「`data.py` の穴埋め粒度」「動作確認例題の具体」から続ければよい。

### (b) 各回が固まったら実装する

1. **正解版を先に作る**（モジュール分割 ＋ 各モジュールの動作確認例題）。
2. kyanite（RTX 2080 Ti）で動作確認。弱 baseline が狙いどおり val 60〜70% / train 70〜80% に収まるか試走で確認。
3. そこから **公開用（解答なし）** を作る。ブランチにするか別リポジトリにするかは未定。
