# 引き継ぎ（これまでの経緯と今後の展望）

## 1. 今の立ち位置

- 勉強会の **運営方針と全 4 回の構成はフィックス済み**（→ `docs/course_design.md`）。
- **各回の地図（トップレベル）は `docs/sessions/index.md` に確定済み**（教員コメント反映・全 4 回・確認スクリプト方針）。
  次は `index.md` を正として、`session0_prep.md` 以降の各回詳細を 1 単位ずつ書く段階。
- 現行リポジトリ（`src/kws/`・`notebooks/`）は **旧方針のまま**（旧 6 回想定・解答公開前提・baseline が test acc≈0.90 と高い）。
  確定方針に沿って **刷新が必要**。

## 2. これまでの経緯（なぜこの大目標・構成・方針になったか）

- **当初案（旧版）**: torchaudio SPEECHCOMMANDS の穴埋め notebook ＋ 解答公開ブランチで「フレームワーク内部の理解」を
  狙う教材だった（→ `discontinued/plan.md`, `discontinued/curriculum.md`）。
- **大目標のピボット**: 大目標を **「PyTorch で動く学習・評価スクリプトを自分で書けるようになること」** に再定義。
  受講者は「ゼロから作る Deep Learning」で理論は履修済みだが実装経験が乏しい、という前提に合わせ、理論の再説明をやめ
  「自分で書く経験」に集中する方針にした。音声認識・torchaudio は題材＝手段と位置づけた。
- **丸写し対策**: 「解答公開 ＋ 穴埋め」だと丸写しで通過できる穴があったため、**解答非公開 ＋ 発表で動作確認例題を実演**に変更。
  詰まったらメンバー同士の相談＋主催者の時間外サポートで吸収する（部分解の配布などはしない）。
- **構成の確定**: 6 観点のサブエージェント（教育設計／運営／受講者／PyTorch 実務／コンペ／批判者）で 2 ラウンド多角評価した。
  最大の論点だった「学習ループ＋評価を 1 回に詰める第 4 回の過密」を、次の 2 点で解消した：
  1. 第 1 回も発表回にして、全回を「前回まとめ（指導者）＋当回内容の予習発表（受講者）→ ハンズオン」の同型イテレーションにする
  2. 学習パートを **実装回（第 3 回）と Tips・深掘り回（第 4 回）に分離**する
- **全 5 回 → 全 4 回への縮約**: 当初は全 5 回（第 5 回をコンペ独立回）で確定したが、時間都合で第 5 回を廃止し、**チューニング大会と leaderboard を第 4 回後半に統合**して全 4 回にした。
- **「動作確認例題」→「確認スクリプト」**: 各モジュールの完成担保を、満たすべき仕様を `assert` で書いた **確認スクリプト `check_*`**（全 PASS で完成）に明文化した。仕様だけ渡し実装本体は含まないので解答非公開と両立する（Karpathy の sanity check を教材化）。
- **発表の意味**: 発表は「前回分の共有」ではなく **「当回内容の予習」**（各自が分担で調べ 1 スライドにまとめ、代表が 1 本だけ発表）に整理した。前回のまとめは指導者が 5 分で行う。
- **baseline**: 既存 baseline は test acc≈0.90 と高く改善余地がないため、**弱 baseline（val 60〜70% / train 70〜80%）** に
  作り替える方針。test リークは教育目的なので許容する。
- **ドキュメント**: 読み手（リード）が異なるので分割管理する（下記）。

## 4. 何を読んで理解するか（この順で）

1. `docs/sessions/index.md` … 各回の地図（全 4 回・確認スクリプト方針）。**各回の具体的内容はこれを正とする**。
2. `docs/course_design.md` … 確定した目的・全 4 回構成・各回の概要・決定事項。
3. `docs/operation.md` … 運営の進め方（1 回のテンプレート・予習発表の運用・巡回・Tips の散らし方）。
4. プロジェクト memory `project_kws-tutorial.md` … 確定方針に至った経緯（解答非公開・大目標の再定義など）。
5. `discontinued/`（`plan.md` / `curriculum.md`）… 旧版。**現行方針ではない**ので参照のみ。

※ `docs/sessions/sessionN_*.md`（各回の超詳細）はまだ存在しない（これから書く）。

## 5. 今後の展望（次にやること）

### (a) 各回の具体的内容を tenk と対話しながら詰める

進め方は **トップレベル → 各回詳細** の 2 段階：

1. `docs/sessions/index.md`（トップレベルの地図）は **確定済み**（全 4 回・確認スクリプト方針）。
2. これを正として、固まった回から `docs/sessions/sessionN_*.md` に **超詳細**を書く（`session0_prep.md` → `session1_*.md` → … の順）。
   ゴールは **「読めば誰でも再現性を持って資料（ノートブック）を作れる」レベル**（セル構成・説明の流れ・コードの骨子・
   確認スクリプトと期待出力＝PASS の見え方まで）。

詰めるべき論点（各 `sessionN_*.md` で）：

- 事前課題・各回で読ませる資料の指定（範囲・「これが言えれば OK」の最低ライン）
- 各回の当日の流れ（セル単位）／当日どこまで実装し、何を宿題に回すか／穴埋めの粒度
- 各モジュールの確認スクリプト `check_*` の assert 内容と PASS の見え方
- 弱ベースラインの具体 config（val 60〜70% / train 70〜80% になる設定。試走で確認する）
- 第 4 回後半 leaderboard の運用・評価指標

進め方は `index.md` の順（事前課題 → 第 1 回 → …）で 1 単位ずつ「調査 → ドラフト → 多角検証 → 提示」で固める。

### (b) 各回が固まったら実装する

1. **正解版を先に作る**（モジュール分割 ＋ 各モジュールの確認スクリプト `check_*`）。
2. kyanite（RTX 2080 Ti）で動作確認。弱 baseline が狙いどおり val 60〜70% / train 70〜80% に収まるか試走で確認。
3. そこから **公開用（解答なし）** を作る。ブランチにするか別リポジトリにするかは未定。
