# 運営計画（勉強会の進め方）

## 目的

受講者は「ゼロから作る Deep Learning」で理論は履修済み．足りないのは **自分で PyTorch のコードを書いた経験**．
本勉強会は，知っている理論を **実際に手を動かして動くコードに落とし込む経験**を積む場とする．
理論の再説明や，フレームワーク内部の細かい仕組みの深掘りはしない（既知のため）．代わりに次を重視する：

- PyTorch の書き方を，自分の手で書けるようになる（`nn.Module` / `DataLoader` / 学習ループ など）
- ゼロから作るでは扱わない **実務スキル**（実験管理 wandb，checkpoint，再現性，計算資源の使い方，エラーの読み方）

## 全体構成

1 モジュール = 1 回．ボトムアップで「動くパイプライン」を積み上げる．
モジュール順: **data → model → 学習ループ → 評価・チューニング**．

- **事前課題**: 環境構築（`uv add` で依存を自分で足す・データ DL）＋ 第 1 回の予習．当日には持ち込まない．
- **各回（同型イテレーション）**: 以下のサイクルを各モジュールで繰り返す．
  1. **前回まとめ**: 指導者／サブ指導者が 5 分でまとめる（受講者にはやらせない）
  2. **予習発表**: 受講者が **当回内容の予習**を分担で 1 スライドにまとめ，代表が 1 本だけ発表
  3. **ハンズオン**: 少し座学 ＋ 前半 md/notebook で説明 → 後半 `src/kws/` の py に実装．確認スクリプトで PASS を取る
  4. **宿題（2 本立て）**: ①実装の続きを `src/kws/` に整理（＋確認スクリプトの PASS 結果＋学び）／②次回の予習（分担で 1 スライド）
- **第 4 回後半**: 学習の深掘りの道具が揃った所でパフォーマンスチューニング大会 → leaderboard 申告まで完結．

回数は **事前課題 ＋ 全 4 回**（第 1 回 キックオフ＋data ／ 第 2 回 model ／ 第 3 回 学習ループ ／ 第 4 回 深掘り＋評価＋チューニング大会）．

## 1 回のテンプレート（90 分目安）

| フェーズ | 時間 | 内容 |
|---|---|---|
| 前回まとめ | 〜5分 | 前回のモジュールを指導者が手短にまとめる（受講者にはやらせない） |
| 予習発表 | 〜15分 | 当回内容の予習を分担で 1 スライドにまとめ，代表が 1 本だけ発表 |
| 少し座学 | 〜15分 | 今日のモジュールが **何を受け取って何を返すか**，PyTorch でどう書くかの見当をつける |
| ハンズオン | 〜45分 | 前半 md/notebook で説明 → 後半 py に実装．確認スクリプトで PASS を取る．**対面で巡回し，詰まりをその場で拾う** |
| まとめ | 〜10分 | 今日の到達確認 ＋ 宿題（実装の続き＋次回の予習）の指示 |

要点: 一番むずかしく価値がある「書く」作業を，**対面（巡回サポートあり）で書き始め**，宿題で仕上げる．輪講の時間内に実装を完結させるのが理想で，早く終わった人と指導者が遅れている人の補助に入る．

## 予習発表（当回内容の予習）

- 内容: 指導者が指定したサイト・観点を分担で調べ，**当回でやる内容の予習**を 1 スライドにまとめる（前回分の共有ではない）．
- 形式: 1 スライド・代表が 10〜15 分で 1 本だけ発表（一人ずつの個別発表はしない）．理論のまとめにはしない（既知のため）．
- ねらい: 当日の内容を予習として頭に入れる ＋ 発表の練習．正確さは指導者がその場でフォローする．
- 発表や院試で忙しい人は予習を軽くしてよいと伝える（重み調整がうまくできていなければ指導者がアドバイスする）．

## 演習に必ず入れる

### 1. パラメータ変更ハンズオン
`lr` / `batch_size` / `epochs` / `n_mels` などを変えて，**学習曲線（`exp/<run>/loss_curve.png` ＋ `history.json`）がどう変わるか**を見る．
「変える前に予想 → 実際に回して確認」の順にすると気づきが大きい．監視は基本この png ＋ json で行い，wandb は第 3 回で名前だけ軽く触れる（必須にしない）．

### 2. 実務 Tips（独立した回にせず，関係する場面に散りばめる）
- **学習を回す回（学習ループ）**: `perf-tracker` で空いている GPU マシンを探す → `tmux` でジョブを投げて回しっぱなしにする → `nvidia-smi` で GPU 使用率・VRAM・誰が使っているかを見る
- **データのダウンロードや前処理の待ち時間**: `htop` で CPU・メモリの様子を見る
- **評価・再現の回**: seed 固定，`uv.lock`，run ディレクトリ規約

## 教材の使い方

- **ノートブック**: 授業中に穴埋めして動かす（前半 md/notebook で説明 → 後半 py に実装）．**解答（`ans`）は公開・案内しない**．
- **確認スクリプト `check_*`**: 各モジュールに，満たすべき仕様を `assert` で書いた確認スクリプトを配る．受講者は実装本体を埋めて **全部 PASS させる**のを目標にする（仕様だけで答えは含まないので解答非公開と両立）．詰まったらメンバー同士の相談＋指導者の時間外サポートで吸収する．
- **`src/kws/`**: 宿題でノートブックの内容を整理して実装する（実装の続き＋確認スクリプトの PASS 結果＋学び）．
- **baseline**: 意図的な **弱ベースライン**（val 60〜70% / train 70〜80%）．高性能化の余地を大きく残し，チューニング大会の出発点にする．
- **各回の内容の詳細**: `docs/sessions/`（地図＝`index.md`，各回詳細＝`sessionN_*.md`）

## 運営の注意

- **環境構築（`uv add` で依存を足す・データダウンロード）は事前課題**にして，本番の時間に持ち込まない（wandb は事前課題では扱わない）
- 対面はとにかく **巡回**して詰まりを拾う（5〜6 名ならこれが一番効くレベル差対策）
- **速い人にはおまけ課題**（例: ConvBlock を 1 段増やすと shape・パラメータ数はどう変わるか）を用意して手持ち無沙汰を防ぐ
- 宿題（実装）が進まない人が出ると次回の発表が崩れるので，**ハンズオンで「動く所まで」を必ず到達**させてから家に送り出す
